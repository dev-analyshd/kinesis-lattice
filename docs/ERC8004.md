# ERC-8004: On-Chain Agent Identity NFT

## Overview

KINESIS uses ERC-8004 to anchor each agent's `did:t3n` identity to an on-chain NFT, providing:
- Verifiable on-chain provenance for agent identities
- Delegation VC signatures that reference an on-chain root
- Cross-chain resolution of agent DIDs from any EVM chain

## How KINESIS Uses ERC-8004

### Agent Spawning

When a KINESIS agent spawns with `initializeT3NIdentity()`:

1. `client.authenticate(createEthAuthInput(address))` resolves the agent's `did:t3n`
2. This DID is linked to an ERC-8004 NFT minted at agent registration
3. The NFT stores: DID, public key, agent metadata, coherence history root

### Delegation VC Binding

Each BBS+ delegation VC issued by KINESIS includes:
```json
{
  "issuer": "did:t3n:testnet:0x...",
  "holder": "did:t3n:testnet:0x...",
  "credentialSubject": {
    "agentIdentity": {
      "erc8004TokenId": "12345",
      "contractAddress": "0x...",
      "chainId": 1,
      "coherenceAtIssuance": 0.87
    }
  }
}
```

### Verification Flow

```
Verifier receives delegation VC
         │
         ↓
1. Resolve did:t3n → ERC-8004 NFT on-chain
         │
         ↓
2. Verify NFT owner = VC issuer's address
         │
         ↓
3. Check T3N revocation registry (inside TEE)
         │
         ↓
4. Verify BBS+ selective disclosure proof
         │
         ↓
5. Accept or reject delegation
```

## Smart Contract Interface

```solidity
interface IERC8004 {
    function agentOf(uint256 tokenId) external view returns (
        string memory did,
        address controller,
        bytes32 coherenceRoot,
        uint64 registeredAt,
        bool revoked
    );

    function mintAgent(
        address to,
        string calldata did,
        bytes32 initialCoherenceRoot
    ) external returns (uint256 tokenId);

    function updateCoherenceRoot(
        uint256 tokenId,
        bytes32 newRoot,
        bytes calldata teeAttestation
    ) external;

    function revoke(uint256 tokenId) external;
}
```

## Integration Status

In KINESIS v0.1.0, ERC-8004 binding is **planned** — the `did:t3n` identity is established via T3N SDK, and the architecture is designed to accept ERC-8004 NFT IDs in delegation VCs once the standard is fully deployed on T3N testnet.
